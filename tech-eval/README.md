This folder contains the data and results for the technical evaluation described in the paper.

1. `data` contains two event log datasets used for evaluation:

- clickstream: E-commerce browsing sessions (10,000 sessions) with events like page_view, add_to_cart, checkout, and purchase. This data is sampled from https://www.kaggle.com/datasets/wafaaelhusseini/e-commerce-transactions-clickstream. We use a sample data since for the automated fact checking evaluation, we needed to update the datasets to the Gemini application to prompt for responses. Large datasets appear to break the Gemini application.
- process: IT incident management tickets (10,000 tickets) with events like ticket creation, assignment, escalation, and resolution. This data is sampled from https://www.kaggle.com/datasets/albertopmd/process-mining-event-log-incident-management.

Each dataset has two CSV files: events.csv (event sequences) and record-attributes.csv (session/ticket-level attributes).

2. `prompts.json` defines the 50 evaluation questions (25 per dataset). 

Each prompt is identified by a combination of dataset (clickstream or process), row operation (none, num-filter, cat-filter, segmentation, pattern-filter), and column operation (numerical, categorical, event-attribute, pattern, funnel).

3. `fact-tagging-codebook.md` defines the rules for tagging ground-truth facts in responses.

4. For `eval1/`, we evaluate the technique's ability to generate insights with sentence-level citations. The pipeline receives a question and dataset, generates an analytic response, and annotates it with `<cite>` tags linking facts to referenced data.

| # | File | Description |
|---|------|-------------|
| 1 | `1. raw-tags.json` | Pipeline output: responses with `<cite>` tags and referenced data |
| 2 | `2. no-tags.json` | Same responses with all tags stripped (plain text) |
| 3 | `3. ground-truth-tags.json` | Human-annotated `<fact>` tags on the plain-text responses |
| 4 | `4. fact-identification-results.json` | Per-fact coverage results (is each ground-truth fact covered by a `<cite>` tag?) |
| 5 | `5. fact-association-results.json` | Per-citation association results (does each `<cite>` tag reference relevant data?) |
| 6 | `6. overall-results.txt` | Summary metrics |

5. For `eval2/`, we evaluate the technique's ability to fact-check existing responses by annotating them with `<cite>` tags (for correct facts) and `<error>` tags (for incorrect facts). The responses being checked were originally generated by Gemini App (`gemini-app-output.txt`).

6. For `eval2/correct/`, the pipeline re-annotates Gemini App responses that contain only correct information.

| # | File | Description |
|---|------|-------------|
| 1 | `1. raw-tags.json` | Pipeline output: re-annotated responses with `<cite>` tags and referenced data |
| 2 | `2. no-tags.json` | Same responses with all tags stripped |
| 3 | `3. ground-truth-tags.json` | Human-annotated `<fact>` tags on the original Gemini App responses |
| 4 | `4. fact-identification-results.json` | Per-fact coverage results |
| 5 | `5. fact-association-results.json` | Per-citation association results |
| 6 | `6. overall-results.txt` | Summary metrics |

7. For `eval2/incorrect/`, the pipeline re-annotates Gemini App responses that contain deliberately introduced errors, using `<error>` tags to flag incorrect facts.

| # | File | Description |
|---|------|-------------|
| 1 | `1. raw-tags.json` | Pipeline output: re-annotated responses with `<cite>` and `<error>` tags and referenced data |
| 2 | `2. no-tags.json` | Same responses with all tags stripped |
| 3 | `3. ground-truth-tags.json` | Ground-truth `<error>` tags from Gemini App's original error annotations |
| 4 | `4. error-identification-results.json` | Per-error coverage results (is each ground-truth error covered by an `<error>` tag?) |
| 5 | `5. error-association-results.json` | Per-error-citation association results (does each `<error>` tag reference relevant data?) |
| 6 | `6. overall-results.txt` | Summary metrics |
